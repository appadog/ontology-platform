# 온톨로지 데이터 플랫폼 — 사용자 가이드 (처음부터 끝까지)

이 문서는 처음 접하는 사용자가 **문서 데이터에서 신뢰할 수 있는 지식 그래프를 만들어 운영**하기까지의 전 과정을 순서대로 안내합니다. 화면 이름·버튼·경로는 실제 제품 기준입니다.

> 대상 독자: 지식 관리자, 데이터 오퍼레이터, 도메인 검수자, 관리자.
> 소요: 첫 프로젝트를 한 바퀴 도는 데 약 20~30분.

---

## 0. 이 플랫폼은 무엇을 하나요

문서(CSV·Excel·PDF·TXT)에서 LLM으로 뽑아낸 지식을 **바로 쓰지 않고**, 후보 단계에서 검증한 뒤 게시해 **근거가 남는 신뢰할 수 있는 지식 그래프**를 만드는 운영 플랫폼입니다.

핵심 3원칙(제품 전반에서 절대 깨지지 않습니다):

1. **후보와 게시를 분리합니다.** 추출 결과는 *후보(candidate) 그래프*에 먼저 쌓이고, 검수를 통과한 항목만 *게시(published) 그래프*로 올라갑니다.
2. **모든 항목에 근거가 남습니다.** 엔티티·관계·속성마다 원천 문서 근거(evidence)와 버전이 연결되어 추적·감사가 가능합니다.
3. **품질을 함께 추적합니다.** 품질 지표, 벤치마크 비교, 학습 인사이트로 추출·검수 품질을 지속적으로 개선합니다.

---

## 1. 화면 구조 이해하기

### 상단 바
- **테넌트 컨텍스트**: 지금 보고 있는 워크스페이스(조직). 전환은 *화면에 보이는 범위만* 바꾸며 데이터를 옮기지 않습니다(읽기 전용·격리).
- **CURRENT PROJECT**: 현재 작업 중인 프로젝트. 대부분의 작업은 프로젝트 안에서 이뤄집니다.
- 우측: 현재 사용자/권한 상태.

### 좌측 내비게이션(LNB) — 작업 흐름 순서대로 묶여 있습니다
- **상단**: `Dashboard`(대시보드), `Projects`(프로젝트), `Admin`(관리).
- **BUILD(구축)**: `Ontology`(온톨로지 모델러) · `Ontology Packs`(온톨로지 팩) · `Sources`(소스) · `Connectors`(커넥터) · `Extraction`(추출) · `Candidates`(후보).
- **REVIEW(검수)**: `Review`(검수 인박스) · `Quality`(품질) · `Governance`(거버넌스).
- **PUBLISH(게시)**: `Publish`(게시 큐) · `Published Graph`(게시 그래프).
- **ANALYZE(활용)**: `Copilot` · `Search`(검색) · `RAG` · `Evaluation`(평가/골드셋) · `Learning Insights`(학습 인사이트) · `Benchmark`(벤치마크) · `External API`(외부 API).

> 이 순서가 곧 사용 흐름입니다: **구축 → 검수 → 게시 → 활용**.

---

## 2. 시작하기 — 대시보드와 프로젝트

1. **대시보드**에서 오늘 무엇을 검수·게시할지 한눈에 확인하고, 상단 카드로 서비스 목적을 파악합니다. `프로젝트 시작하기` 또는 `최근 프로젝트 열기`로 진입합니다.
2. **Projects**에서 프로젝트를 만들거나 선택합니다. 상단 `CURRENT PROJECT`에서 언제든 전환할 수 있습니다.
3. **프로젝트 홈**에는 요약(소스 수·온톨로지 버전·최근 변경)과 **워크플로 스테퍼**(`Project → Ontology → Source → Extraction → Candidates → Evidence`)가 있어 다음 할 일을 알려줍니다. "현재/다음/준비" 상태로 진행 위치를 표시합니다.

---

## 3. 1단계 · 온톨로지 설계 (BUILD → Ontology)

지식을 담을 **구조**를 먼저 만듭니다. 온톨로지는 항상 **DRAFT(초안)** 상태에서 편집합니다.

1. `Ontology` 진입 → 상단 **Authoring actions**에서 순서대로 정의합니다.
   - **Class(클래스) 생성**: 예) `Company`, `부서`, `정책`.
   - **Property(속성) 추가**: 대상 클래스에 이름·데이터 타입(STRING 등)·카디널리티(OPTIONAL/필수 등) 지정. 예) `company_name : STRING`.
   - **Relation(관계) 생성**: 출발 클래스 → 대상 클래스, 관계 카디널리티(1:1 / 1:N / N:N) 지정. 예) `정책 —(owned by)→ 부서`.
2. 좌측 **Classes** 목록과 우측 **그래프 캔버스**로 구조를 확인합니다. 검색으로 특정 클래스를 찾습니다.
3. 클래스/속성/관계는 **추가·수정·삭제**가 가능합니다. 클래스 삭제는 소프트 삭제(연결된 속성·관계도 함께 `DELETED` 처리)로 안전하게 동작합니다.

> 팁: 완벽할 필요는 없습니다. 핵심 클래스와 관계만 먼저 잡고 추출 결과를 보며 다듬으면 됩니다.

---

## 4. 2단계 · 소스 준비 (BUILD → Sources)

추출의 원천이 될 문서를 올립니다.

1. `Sources` → **원천 데이터 업로드**에서 파일을 선택하고 **SOURCE TYPE**(CSV/Excel/PDF/TXT)과 표시 이름을 지정해 업로드합니다.
2. 업로드된 소스는 **STATUS**(예: `PROFILED · 프로파일 완료`)와 **다음 단계**(예: `Profile`)를 표에 표시합니다.
3. 소스를 열어 **프로파일**(컬럼/구조 파악)과 **청크(chunks)**(구간 보기)를 확인합니다. 여기서 데이터 품질·구조를 미리 점검하면 추출 정확도가 올라갑니다.

> (선택) **Connectors**로 외부 시스템의 데이터를 미리보기(dry-run)할 수 있습니다. 커넥터는 읽기 전용 미리보기이며 아무것도 가져오거나 실행하지 않습니다.

---

## 5. 3단계 · 추출 실행 (BUILD → Extraction)

온톨로지 구조에 맞춰 소스에서 후보 지식을 뽑아냅니다.

1. `Extraction` → **새 추출**에서 대상 소스와 프롬프트/모델 설정을 고르고 추출 잡을 실행합니다.
2. `Extraction Jobs`에서 잡 상태와 모델 실행(`model_runs`)을 추적합니다. 재시도 체인은 중복 없이 정리되어 표시됩니다.
3. 추출 결과는 **후보(candidate) 그래프**로 쌓입니다 — 게시 그래프에는 아직 영향이 없습니다.

---

## 6. 4단계 · 후보와 근거 확인 (BUILD → Candidates)

1. `Candidates`에서 추출된 후보 엔티티·관계·속성을 살펴봅니다.
2. 각 후보의 **Evidence(근거)** 를 열어 어떤 소스·구간에서 나왔는지 원문 맥락과 함께 확인합니다. 근거가 끊긴 경우에도 소스/잡/후보 맥락과 복구 동선이 제공됩니다.
3. 여기서 후보의 신뢰도를 가늠하고, 본격 검수(Review)로 넘어갑니다.

---

## 7. 5단계 · 검수 (REVIEW → Review 인박스 → Workbench)

후보를 사람이 검증해 게시 여부를 결정합니다. **검수 결정은 게시 사실과 분리되어**, 게시 자격을 통과하기 전까지 게시 그래프에 반영되지 않습니다.

1. `Review`(검수 인박스)에서 **큐 필터**(담당/상태/검증/신뢰도)로 볼 항목을 좁힙니다. 각 태스크는 우선순위·사유를 미리 보여줍니다.
2. 태스크를 열어 **Workbench(검수 워크벤치)** 에서 후보를 검증합니다.
   - 검증(validation) 결과·심각도를 확인하고, **승인/수정/반려**를 결정합니다.
   - 근거 우선(evidence-first) 순서로 원문을 먼저 읽고 판단하도록 구성되어 있습니다.
3. 검수를 통과한 항목은 **게시 자격(publish eligibility)** 을 얻습니다. 자격 사유 코드로 왜 게시 가능/불가한지 알 수 있습니다.

> 하위 탭(`Review inbox | Workbench | Publish queue | Published graph | Quality`)으로 검수→게시 흐름을 오갈 수 있습니다.

---

## 8. 6단계 · 게시 (PUBLISH → Publish 큐 → Published Graph)

1. `Publish`(게시 큐)에서 게시 자격을 통과한 항목을 확인하고 **게시 잡(publish job)** 을 실행합니다.
2. 게시 잡 상세에서 진행 상황과 결과(버전 승격, 계보)를 추적합니다.
3. `Published Graph`(게시 그래프 탐색기)에서 **게시된 사실만** 탐색합니다.
   - **탐색기 탭**: 루트 중심 이웃 뷰(홉 depth 조절, 최대 3홉). 그래프가 너무 크면 안전하게 요약만 보여주고 부분 그래프를 억지로 그리지 않습니다.
   - **시각화 · 요약 탭**: 읽기 전용 전체 그래프 시각화 + 요약 통계(노드/엣지 수, 클래스·관계별 분포, 밀도, 컴포넌트). **읽기 전용이라 그래프를 변경/게시/저장하지 않으며**, 화면에 6개 변경 가드 플래그가 모두 false임을 증명해 보여줍니다.

> 후보/게시 분리 원칙 덕분에, 게시 그래프에는 검증된 사실만 근거와 함께 남습니다.

---

## 9. 7단계 · 품질 추적 (REVIEW → Quality)

1. `Quality`(품질 대시보드)에서 게시·검수 품질 지표를 확인합니다.
2. 설명 가능한 지표 그룹으로 어디를 개선해야 할지 파악합니다(가중 종합 점수·협업/SLA는 후속 범위).
3. 품질 신호를 다음 라운드의 온톨로지·프롬프트·검수 기준 개선에 반영합니다.

---

## 10. 8단계 · 거버넌스 (REVIEW → Governance)

게시 그래프에 대한 변경을 통제된 절차로 관리합니다.

1. `Governance`에서 **변경 요청(change request)** 을 생성합니다.
2. **영향 시뮬레이션(impact)** 으로 이 변경이 무엇에 영향을 주는지 미리(dry-run) 확인합니다 — 실제로는 아무것도 바꾸지 않습니다.
3. 승인 정책에 따라 검토·승인 후 변경이 적용됩니다. 모든 변경은 근거·버전과 함께 감사됩니다.

---

## 11. 활용 (ANALYZE)

게시된 지식 그래프를 실제로 사용하는 단계입니다. 검색·질의는 **읽기 전용**이며, 후보 그래프 사실은 답변/인용에서 제외됩니다.

- **Search(검색)**: 게시 그래프에서 엔티티·관계를 검색합니다.
- **RAG**: 게시된 사실을 근거로 질의응답합니다(후보 사실은 인용하지 않음).
- **Copilot**: 온톨로지·검수 작업을 돕는 제안. 제안은 미리보기이며 사용자가 승인하기 전까지 아무것도 바꾸지 않습니다.
- **Learning Insights(학습 인사이트)**: 반복되는 검수 패턴을 학습해 자동 승인 후보 등을 제안(미리보기).
- **Evaluation / Gold Set(평가/골드셋)**: 정답 셋을 만들고 버전 관리해 추출·검수 품질을 정량 평가합니다.
- **Benchmark(벤치마크)**: 실행/모델 간 성능을 비교(컨퓨전 매트릭스 등)합니다.
- **Ontology Packs(온톨로지 팩)**: 미리 준비된 온톨로지 번들을 카탈로그에서 보고, **적용 미리보기(dry-run)** 로 무엇이 추가/충돌/중복될지 확인합니다 — 실제 설치·적용은 하지 않습니다.
- **External API(외부 API)**: 읽기 전용 외부 연동.

---

## 12. 관리 (Admin)

프로젝트/조직 단위로 역할·권한, 자격증명(마스킹), 승인 정책, 가져오기/내보내기, 운영, 보존/백업을 관리합니다. 조직 관리 화면은 조직 관리자만 접근합니다.

> 내부 데이터·자료를 업로드할 때는 사내 절차에 따라 부서장과 협의하세요.

---

## 13. 전체 흐름 요약 (한 장)

```text
① 온톨로지 설계(DRAFT)        BUILD → Ontology
        ↓ 구조 확정
② 소스 준비(업로드·프로파일)   BUILD → Sources  (선택: Connectors 미리보기)
        ↓
③ 추출 실행 → 후보 생성        BUILD → Extraction → Candidates(+Evidence 근거)
        ↓  (후보 그래프에만 쌓임)
④ 검수(승인/수정/반려)         REVIEW → Review 인박스 → Workbench
        ↓  게시 자격 통과
⑤ 게시(게시 잡 실행)           PUBLISH → Publish 큐 → Published Graph(탐색기/시각화·요약)
        ↓  (게시 그래프 = 검증된 사실 + 근거)
⑥ 품질·거버넌스로 개선/통제     REVIEW → Quality / Governance(+영향 시뮬레이션)
        ↓
⑦ 활용                         ANALYZE → Search / RAG / Copilot / Benchmark / Gold Set …
        ↺ 반복하며 그래프를 키우고 품질을 높임
```

## 14. 자주 막히는 지점
- **"추출했는데 게시 그래프에 안 보여요."** 정상입니다. 후보는 검수를 통과해야 게시됩니다(⑦ 아님, ④→⑤ 순서).
- **"게시 그래프가 너무 커서 안 그려져요."** 안전 장치입니다. 요약만 표시되며, 필터로 범위를 좁혀 보세요.
- **"온톨로지를 바꿔도 게시 그래프가 그대로예요."** 온톨로지는 DRAFT에서 편집되고, 게시 그래프는 게시 절차를 거쳐야 반영됩니다.
- **"커넥터/팩/코파일럿을 눌러도 아무것도 안 바뀌어요."** 이들은 읽기 전용 미리보기입니다. 실제 반영은 각자의 정식 경로(온톨로지 편집·게시·거버넌스)를 따릅니다.

---

이 가이드는 실제 화면(대시보드·프로젝트 홈·온톨로지 모델러·소스·검수 인박스·게시 그래프)을 따라 작성되었습니다. 화면 문구는 지속적으로 다듬어지므로, 라벨이 조금 달라도 위 **구축 → 검수 → 게시 → 활용** 흐름은 동일합니다.
