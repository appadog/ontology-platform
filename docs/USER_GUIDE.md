# 온톨로지 데이터 플랫폼 — 사용자 가이드 (처음부터 끝까지)

문서(CSV·Excel·PDF·TXT)에서 **신뢰할 수 있는 지식 그래프**를 만들어 운영하기까지, 처음 쓰는 사용자가 따라 할 수 있도록 단계별로 안내합니다. 화면 이름·버튼·상태 값은 실제 제품 기준입니다.

- **가장 빠른 길**: [3. 20분 따라하기](#3-20분-따라하기-첫-게시까지)만 그대로 따라 하면 첫 게시까지 끝납니다.
- **개념부터**: [1. 5분 개념](#1-5분-개념-이-플랫폼이-하는-일)  ·  **화면 지도**: [2. 화면 구조](#2-화면-구조-한눈에)  ·  **막힐 때**: [12. 문제 해결](#12-자주-막히는-지점--문제-해결)

## 1. 5분 개념 — 이 플랫폼이 하는 일

LLM이 문서에서 뽑은 지식을 **바로 쓰지 않습니다.** 먼저 *후보*로 쌓고, 사람이 검수한 것만 *게시*합니다.

```text
 문서            LLM 추출         후보 그래프            검수            게시 그래프
(원천 소스) ─────────────▶  (candidate, 임시)  ──────────▶  (published, 확정)
                                  │  아직 아무 데도 안 쓰임        │  근거와 함께 확정 저장
                                  └─ 사람이 승인/수정/반려 ────────┘
```

**절대 바뀌지 않는 3원칙** — 이걸 알면 제품 전체가 이해됩니다.

| 원칙 | 뜻 | 사용자에게 주는 이점 |
|---|---|---|
| **후보 ≠ 게시** | 추출 결과는 후보에만 쌓이고, 검수 통과분만 게시된다 | 검증 안 된 정보가 지식 그래프를 오염시키지 않음 |
| **모든 항목에 근거** | 엔티티·관계·속성마다 원천 문서 evidence + 버전이 붙는다 | "이 사실은 어디서 왔나?"를 항상 추적·감사 가능 |
| **품질을 추적** | 완전성·일관성·추적성·검증 통과율을 지표로 관리 | 어디를 고쳐야 품질이 오르는지 보임 |

> 그리고 커넥터·팩·코파일럿·검색·RAG 등 **활용 도구는 전부 읽기 전용(미리보기)** 입니다. 실제 변경은 언제나 정식 경로(온톨로지 편집 → 검수 → 게시 → 거버넌스)로만 일어납니다.

## 2. 화면 구조 한눈에

### 상단 바
- **테넌트 컨텍스트** — 지금 보는 워크스페이스(조직). 전환은 *보이는 범위만* 바꾸며 데이터를 옮기지 않습니다.
- **CURRENT PROJECT** — 현재 작업 중인 프로젝트. 거의 모든 작업은 이 프로젝트 안에서 이뤄집니다. 여기서 언제든 전환합니다.
- 우측 — 현재 사용자/권한(예: `dev-admin`).

### 좌측 내비게이션(LNB) = 작업 순서
왼쪽 메뉴는 **작업 흐름 순서대로** 묶여 있습니다. 위에서 아래로 내려가면 그게 곧 사용법입니다.

| 그룹 | 메뉴 | 언제 쓰나 |
|---|---|---|
| (상단) | Dashboard · Projects · Admin | 시작·프로젝트 선택·관리 |
| **BUILD 구축** | Ontology · Ontology Packs · Sources · Connectors · Extraction · Candidates | ①구조 설계 ②문서 준비 ③추출 ④후보 확인 |
| **REVIEW 검수** | Review · Quality · Governance | ⑤검수 ·품질·변경통제 |
| **PUBLISH 게시** | Publish · Published Graph | ⑥게시·게시 그래프 탐색 |
| **ANALYZE 활용** | Copilot · Search · RAG · Evaluation · Learning Insights · Benchmark · External API | 게시된 지식 활용·품질 개선 |

> **모바일**: 상단 앱바의 **☰(햄버거)** 를 눌러 메뉴를 엽니다. 페이지를 이동하면 메뉴는 자동으로 닫힙니다.

### 어느 화면에서나 있는 것
- **브레드크럼**(예: `기업 문서 지식그래프 / 검수`) — 현재 위치. 클릭해 상위로 이동.
- **워크플로 스테퍼** — `Project → Ontology → Source → Extraction → Candidates → Evidence`를 `완료 / 현재 / 다음 / 준비`로 표시해 다음 할 일을 알려줍니다.
- **상태 배지** — `값 · 한글 설명` 형태(예: `READY · 준비됨`, `PENDING · 대기`). 읽는 법은 [13장](#13-상태-배지사유-코드-읽는-법).

## 3. 20분 따라하기 — 첫 게시까지

데모 프로젝트로 전 과정을 한 바퀴 돕니다. 각 단계에 **"이게 보이면 성공"** 신호를 넣었습니다.

> 준비: 브라우저에서 플랫폼 접속 → 상단 `CURRENT PROJECT`에서 **기업 문서 지식그래프**(또는 임의 데모 프로젝트) 선택.

1. **구조 확인** — LNB `BUILD → Ontology`. `정책`, `부서` 같은 클래스와 관계가 보이면 구조가 이미 있는 것. (없으면 [4단계](#4-1단계--온톨로지-설계)에서 만듭니다.)
   ✅ *성공: 왼쪽 Classes 목록에 클래스가 1개 이상.*
2. **문서 확인** — `BUILD → Sources`. 표에 `policy-index.csv` 등이 `PROFILED · 프로파일 완료 / READY · 준비됨`으로 보이면 소스 준비 완료.
   ✅ *성공: STATUS가 `PROFILED`/`READY`.*
3. **추출 실행** — `BUILD → Extraction → 추출 작업 만들기`. `SOURCE`, `ONTOLOGY VERSION`(예: `v1 DRAFT`), `PROMPT`, `FIXTURE`(예: `Success`)를 고르고 **`추출 작업 만들기`** 클릭.
   ✅ *성공: 추출 작업 목록에 `SUCCESS · 성공` 행이 생기고 `CANDIDATES 보기` 링크가 나타남.*
4. **후보 확인** — 그 행의 **`보기`** → 후보 목록. 각 후보의 **근거(Evidence)** 를 열어 어느 문서·구간에서 나왔는지 확인.
   ✅ *성공: 후보에 `PRESENT · 있음` 근거가 붙어 있음.*
5. **검수** — `REVIEW → Review`(검수 인박스). `검수 큐 필터`에서 `담당: 나에게 배정됨` 등으로 좁히고, 태스크를 열어 **워크벤치**에서 근거를 읽고 **승인/수정/반려** 결정.
   ✅ *성공: 태스크 상태가 `APPROVED · 승인됨`(또는 `MODIFIED · 수정 승인`)으로 바뀜.*
6. **게시** — `PUBLISH → Publish`(게시 큐). `ELIGIBILITY`가 **`ELIGIBLE · 게시 가능`** 인 후보를 선택(`Select eligible`) → **`Prepare publish`(게시 준비)** → 게시 잡 실행.
   ✅ *성공: 게시 잡이 완료되고 버전이 올라감.*
7. **결과 확인** — `PUBLISH → Published Graph`. **탐색기** 탭에서 게시된 사실을, **시각화 · 요약** 탭에서 전체 그래프와 통계를 확인.
   ✅ *성공: `READY · 준비됨`, "Published graph v_N_ current", 요약 통계가 보임.*

여기까지가 한 사이클입니다. 이후에는 소스를 늘리고 검수·게시를 반복하며 그래프를 키웁니다.

## 4. 1단계 · 온톨로지 설계  (LNB: BUILD → Ontology)

지식을 담을 **구조(클래스·속성·관계)** 를 만듭니다. 온톨로지는 항상 **DRAFT(초안)** 에서 편집합니다(우측 상단 `DRAFT · 초안` 배지).

**Authoring actions** 패널에서 순서대로:

| 만들 것 | 입력 | 예시 |
|---|---|---|
| **클래스** | 이름 → `Create class` | `정책`, `부서`, `Company` |
| **속성** | 클래스에 속성명·`DATA TYPE`(STRING 등)·`PROPERTY CARDINALITY`(OPTIONAL 등) → `Create property` | `company_name : STRING · OPTIONAL` |
| **관계** | `RELATION`·`TARGET CLASS`·`RELATION CARDINALITY`(1:1 / 1:N / N:N) → `Create relation` | `정책 —(owned by)→ 부서`, `N:N` |

- 좌측 **Classes** 목록 + 우측 **그래프 캔버스**로 구조를 확인합니다(검색으로 특정 클래스 찾기).
- 수정·삭제 가능. 클래스 삭제는 **소프트 삭제**(연결된 속성·관계도 함께 정리)라 안전합니다.

✅ **성공 신호**: Classes 목록에 원하는 클래스가 있고, 캔버스에 관계 선이 그려짐.
💡 **팁**: 완벽할 필요 없습니다. 핵심 클래스·관계만 먼저 잡고, 추출 결과를 보며 다듬으세요.

## 5. 2단계 · 소스 준비  (LNB: BUILD → Sources)

추출의 원천 문서를 올립니다.

1. **원천 데이터 업로드** 카드에서 **`파일 선택`** → 파일 지정 → `SOURCE TYPE`(CSV/Excel/PDF/TXT)·표시 이름 → **`소스 업로드`**.
   > 상단 버튼은 파일이 없으면 `파일 선택`으로 인라인 폼을 열어 주고, 파일을 고르면 `소스 업로드`로 바뀝니다.
2. 업로드되면 **원천 데이터 목록** 표에 `FILE / TYPE / STATUS / PREVIEW / NEXT / SIZE / UPLOADED`로 나타납니다.
3. 소스를 열어 **프로파일**(컬럼/구조)과 **청크**(구간 보기)를 확인합니다. 여기서 품질을 미리 점검하면 추출 정확도가 올라갑니다.

✅ **성공 신호**: STATUS `PROFILED · 프로파일 완료`, PREVIEW `READY · 준비됨`, NEXT `Profile`.
🔒 **주의**: 내부 데이터·자료 업로드 시 사내 절차에 따라 **부서장과 협의**하세요.

## 6. 3단계 · 추출 실행  (LNB: BUILD → Extraction)

온톨로지 구조에 맞춰 소스에서 후보 지식을 뽑습니다.

1. **`추출 작업 만들기`** → **추출 입력**에서 선택:
   - `SOURCE` (예: `policy-index.csv`)
   - `ONTOLOGY VERSION` (예: `v1 DRAFT`)
   - `PROMPT` + `프롬프트 버전`(예: `v1 ACTIVE`)
   - `PROVIDER`/`MODEL` (데모는 `MockProvider`)
   - `FIXTURE` — 재현 시나리오: `Success`(정상) / `Partial failure`(부분 실패) / `Broken evidence`(근거 손상) / `Missing fixture`.
2. 우측 **선택한 맥락**에서 고른 값을 확인하고 **`추출 작업 만들기`** 클릭.
3. **추출 작업 목록**에서 상태를 추적: `SUCCESS · 성공`, `PARTIAL_FAILED · 부분 실패` 등. 각 행의 **`보기`** 로 후보로 이동.

✅ **성공 신호**: 목록에 `100%` + `SUCCESS · 성공` 행 + `CANDIDATES 보기` 링크.
💡 **핵심**: 추출 결과는 **후보 그래프에만** 쌓입니다. 게시 그래프는 아직 그대로입니다.

---

## 7. 4단계 · 후보와 근거 확인  (LNB: BUILD → Candidates)

1. 후보 엔티티·관계·속성을 살펴봅니다(추출 잡의 `보기`로도 진입).
2. 각 후보의 **Evidence(근거)** 를 열어 어떤 소스·구간에서 나왔는지 **원문 맥락**과 함께 확인합니다.
3. 근거가 끊긴 경우에도 소스/잡/후보 맥락과 복구 동선이 제공됩니다.

✅ **성공 신호**: 후보에 근거 `PRESENT · 있음`이 붙고 원문을 열어볼 수 있음.

---

## 8. 5단계 · 검수  (LNB: REVIEW → Review)

후보를 사람이 검증해 게시 여부를 결정합니다. **검수 결정은 게시 사실과 분리**되어, 게시 자격을 통과하기 전까지 게시 그래프에 반영되지 않습니다.

### 8-1. 검수 인박스
- **검수 큐 필터**: `담당`(나에게 배정됨/미배정/전체) · `상태`(대기/승인됨/반려됨/수정) · `검증`(통과/경고/실패) · `신뢰도`.
- 각 태스크는 열기 전에 **우선순위·사유**를 미리 보여줍니다.
- 상단 하위 탭으로 `검수 인박스 | 검수 워크벤치 | 게시 큐 | 게시 그래프 | 품질`을 오갑니다.

### 8-2. 검수 워크벤치 (태스크 열기)
- **검증(validation) 결과·심각도**를 확인합니다: `PASSED · 통과` / `WARNING · 경고` / `FAILED · 실패`.
- **근거 우선(evidence-first)**: 원문을 먼저 읽고 판단하도록 배치.
- 결정: **승인 / 수정 / 반려**. 경고를 남길 때는 **사유**를 함께 적습니다(사유 없으면 나중에 게시가 막힙니다 — [13장](#13-상태-배지사유-코드-읽는-법)).

✅ **성공 신호**: 태스크 상태가 `APPROVED · 승인됨` 또는 `MODIFIED · 수정 승인`.

## 9. 6단계 · 게시  (LNB: PUBLISH → Publish → Published Graph)

### 9-1. 게시 큐
검수를 통과한 후보 중 **게시 자격(eligibility)** 을 갖춘 것만 게시합니다. 표에서 각 후보의 `REVIEW / VALIDATION / EVIDENCE / ELIGIBILITY`를 확인하고:

1. `ELIGIBILITY = ELIGIBLE · 게시 가능`인 후보를 선택(`Select eligible`).
2. **`Prepare publish`(게시 준비)** → 게시 잡 실행.

게시가 막히는 후보는 사유 코드로 이유를 알려줍니다 — 예: `WARNING_REASON_REQUIRED · 경고 사유 필요`, `FAILED_VALIDATION · 검증 실패`, `MISSING_EVIDENCE · 근거 없음`, `NEEDS_DISCUSSION · 논의 필요`. (전체 표는 [13장](#13-상태-배지사유-코드-읽는-법).)

### 9-2. 게시 잡 확인
게시 잡 상세에서 진행 상황과 결과(버전 승격·계보)를 추적합니다.

### 9-3. 게시 그래프 탐색  (Published Graph)
게시된 **사실만** 탐색합니다. 두 개의 탭:

- **탐색기** — 루트 중심 이웃 뷰. 홉(hop) depth 조절(기본 2, 최대 3). 그래프가 너무 크면 **안전하게 요약만** 보여주고 부분 그래프를 억지로 그리지 않습니다.
- **시각화 · 요약** — 읽기 전용 전체 그래프 + 요약 통계(노드/엣지 수, 클래스·관계별 분포, 밀도, 컴포넌트). **읽기 전용**이라 그래프를 변경/게시/저장하지 않으며, 화면에 "**6개 mutation 플래그 모두 false**" 증명을 표시합니다.

✅ **성공 신호**: `READY · 준비됨`, "Published graph v_N_ current", 요약 통계 표시.

---

## 10. 7단계 · 품질 / 거버넌스

### 10-1. 품질 대시보드  (REVIEW → Quality)
게시 그래프의 품질을 **설명 가능한 지표**로 봅니다.

| 지표 그룹 | 뜻 |
|---|---|
| 완전성 (COMPLETENESS) | 필수 속성·근거를 갖춘 사실의 비율 |
| 일관성 (CONSISTENCY) | 서로 모순되지 않는 정도 |
| 추적성 (TRACEABILITY) | 근거·버전으로 되짚을 수 있는 정도 |
| 검증 통과율 (VALIDATION) | 통과 N · 경고 N · 실패 N |

- 각 지표는 **분자/분모·Formula ID·드릴다운**까지 열어볼 수 있어 "왜 이 수치인가"를 설명합니다.
- 버전 선택기로 현재/과거 게시 버전을 비교합니다. (가중 종합 점수는 후속 범위 — `NO COMPOSITE SCORE`.)

### 10-2. 거버넌스  (REVIEW → Governance)
게시 그래프에 대한 변경을 **통제된 절차**로 관리합니다.

1. **`변경 요청 생성`** 으로 변경 요청(change request)을 만듭니다.
2. **영향 시뮬레이션(impact)** 으로 무엇에 영향을 주는지 **미리(dry-run)** 확인 — 실제로는 아무것도 바꾸지 않습니다.
3. 상태 흐름: `DRAFT · 초안` → `OPEN · 검토 대기` → `IN_REVIEW · 검토 중` → `APPROVED · 승인됨` / `REJECTED · 반려됨` / `WITHDRAWN · 철회됨`.
   - **중요**: 승인은 **큐잉된 의도**입니다(`QUEUED · 큐잉됨(미적용)`). 온톨로지·게시 그래프는 그 즉시 바뀌지 않고, 실제 적용은 이후 사람이 개시하는 별도 단계에서 감사와 함께 이뤄집니다.

---

## 11. 활용 도구 (ANALYZE) — 모두 안전(읽기 전용)

게시된 지식을 실제로 쓰는 단계. 질의는 **읽기 전용**이며, 후보 사실은 답변/인용에서 제외됩니다.

| 도구 | 하는 일 | 안전장치 |
|---|---|---|
| **Search** 검색 | 게시 그래프에서 엔티티·관계 검색 | 읽기 전용 |
| **RAG** | 게시된 사실을 근거로 질의응답 | 후보 사실 인용 안 함 |
| **Copilot** 코파일럿 | 검수·거버넌스 작업 제안 | **제안 전용** — `NO_AUTO_APPLY/PUBLISH/APPROVE`, 실제 LLM 호출 없음(결정적 mock). 채택해도 기존 사람 검토 게이트로 이동만 함 |
| **Evaluation / Gold Set** | 정답 셋을 만들고 버전 관리해 품질 정량 평가 | 평가용 |
| **Learning Insights** | 반복 검수 패턴 학습 → 자동 승인 후보 등 제안 | 미리보기 |
| **Benchmark** | 실행/모델 간 성능 비교(컨퓨전 매트릭스 등) | 읽기 전용 |
| **Ontology Packs** | 준비된 온톨로지 번들 카탈로그 + **적용 미리보기(dry-run)** | 실제 설치/적용 안 함 |
| **Connectors** | 외부 시스템 데이터 **미리보기(dry-run)** | 가져오기/실행 안 함 |
| **External API** | 읽기 전용 외부 연동 | 읽기 전용 |

> 공통 규칙: 이 도구들은 무엇을 눌러도 **그래프를 바꾸지 않습니다.** 실제 변경은 온톨로지 편집 → 검수 → 게시 → 거버넌스로만 일어납니다. 화면의 "mutation 플래그 모두 false" 표시가 그 증거입니다.

## 12. 자주 막히는 지점 · 문제 해결

| 증상 | 원인 | 해결 |
|---|---|---|
| 추출했는데 게시 그래프에 안 보임 | 정상. 후보는 검수를 통과해야 게시됨 | ⑤검수 → ⑥게시 순서로 진행 |
| 게시 큐에서 후보가 게시 안 됨 | 게시 자격 미충족 | `ELIGIBILITY` 사유 코드 확인 → [13장](#13-상태-배지사유-코드-읽는-법) 표대로 조치 |
| "경고 사유 필요"로 막힘 | 검수에서 경고를 남기며 **사유**를 안 적음 | 워크벤치로 돌아가 경고 사유 입력 |
| "근거 없음"으로 막힘 | 후보에 evidence가 없음 | 근거 있는 추출로 다시 만들거나 반려 |
| 게시 그래프가 안 그려짐 | 그래프가 너무 큼(안전장치) | 요약만 표시됨. 탐색기에서 홉/필터로 범위 축소 |
| 온톨로지를 바꿔도 게시 그래프 그대로 | 온톨로지는 DRAFT에서 편집, 게시 그래프는 게시 절차 필요 | 변경을 검수·게시(또는 거버넌스)로 반영 |
| 커넥터/팩/코파일럿을 눌러도 안 바뀜 | 전부 읽기 전용 미리보기 | 정상. 실제 반영은 정식 경로로 |
| 승인했는데 적용이 안 됨(거버넌스) | 승인=큐잉된 의도(`QUEUED·미적용`) | 이후 사람이 개시하는 적용 단계 필요 |
| 모바일에서 메뉴가 안 보임 | 메뉴가 접혀 있음 | 상단 **☰** 탭 |

## 13. 상태 배지·사유 코드 읽는 법

배지는 **`값 · 한글 설명`** 형태입니다(예: `READY · 준비됨`). 자주 보는 값:

**검수/후보 상태** — `PENDING · 대기` / `APPROVED · 승인됨` / `MODIFIED · 수정 승인` / `REJECTED · 반려됨` / `NEEDS_DISCUSSION · 논의 필요`
**검증** — `PASSED · 통과` / `WARNING · 경고` / `FAILED · 실패`
**근거** — `PRESENT · 있음` / `MISSING · 없음`
**게시 여부** — `NOT_PUBLISHED · 미게시` / `PUBLISHED · 게시됨`

**게시 자격(ELIGIBILITY) 사유 코드** — 게시 큐에서 게시 가능/불가를 알려줍니다.

| 코드 | 뜻 | 조치 |
|---|---|---|
| `ELIGIBLE · 게시 가능` | 바로 게시 가능 | 선택 후 게시 |
| `PENDING · 대기` | 아직 검수 미완 | 검수 완료 |
| `WARNING_REASON_REQUIRED · 경고 사유 필요` | 경고인데 사유 미기재 | 워크벤치에서 사유 입력 |
| `FAILED_VALIDATION · 검증 실패` | 검증 실패 | 수정 후 재추출/재검수 |
| `MISSING_EVIDENCE · 근거 없음` | 근거 없음 | 근거 확보 또는 반려 |
| `NEEDS_DISCUSSION · 논의 필요` | 논의 필요로 표시됨 | 합의 후 재검수 |

**거버넌스 상태** — `DRAFT · 초안` → `OPEN · 검토 대기` → `IN_REVIEW · 검토 중` → `APPROVED · 승인됨`(`QUEUED · 큐잉됨(미적용)`) / `REJECTED · 반려됨` / `WITHDRAWN · 철회됨`

## 14. 용어 사전

| 용어 | 뜻 |
|---|---|
| **온톨로지(Ontology)** | 지식을 담는 구조 — 클래스·속성·관계의 정의 |
| **클래스 / 속성 / 관계** | 개체 종류 / 개체의 특성 / 개체 간 연결 |
| **카디널리티** | 관계·속성의 개수 규칙(1:1, 1:N, N:N / 필수·선택) |
| **DRAFT(초안) / PUBLISHED(게시)** | 편집 중 구조 / 확정·게시된 구조 |
| **소스(Source)** | 추출의 원천 문서(CSV·Excel·PDF·TXT) |
| **프로파일 / 청크** | 소스의 컬럼·구조 파악 / 구간 단위 분할 |
| **추출(Extraction)** | 소스에서 온톨로지에 맞는 후보를 뽑는 작업 |
| **후보(Candidate)** | 추출된 임시 지식(아직 미게시) |
| **근거(Evidence)** | 후보/사실이 나온 원천 문서 위치·맥락 |
| **검수(Review)** | 후보를 사람이 검증해 승인/수정/반려하는 단계 |
| **게시 자격(Eligibility)** | 후보가 게시될 수 있는지의 판정과 사유 |
| **게시 그래프(Published Graph)** | 검수를 통과해 확정된 지식 그래프 |
| **홉(hop)** | 그래프에서 한 노드로부터 떨어진 관계 단계 수 |
| **거버넌스 / 영향 시뮬레이션** | 통제된 변경 절차 / 변경 영향을 미리 보는 dry-run |
| **dry-run / mutation 가드** | 아무것도 바꾸지 않는 미리보기 / "안 바꿨음"을 증명하는 플래그 |

## 15. 역할별 빠른 길잡이

| 역할 | 주로 쓰는 곳 | 핵심 할 일 |
|---|---|---|
| **데이터 오퍼레이터** | BUILD(Ontology·Sources·Extraction·Candidates) | 구조 잡고 문서 올려 후보 만들기 |
| **도메인 검수자** | REVIEW(Review·Quality) | 근거 보고 승인/수정/반려, 품질 확인 |
| **게시 담당** | PUBLISH(Publish·Published Graph) | 자격 통과분 게시, 게시 그래프 점검 |
| **거버넌스/관리자** | Governance · Admin | 변경 요청 통제, 권한·정책·백업 관리 |
| **활용자** | ANALYZE(Search·RAG·Benchmark 등) | 게시된 지식 검색·질의·평가 |

---

### 한 장 요약
```text
① 온톨로지 설계(DRAFT)     BUILD → Ontology
② 소스 준비(업로드·프로파일) BUILD → Sources     (선택: Connectors 미리보기)
③ 추출 → 후보 생성          BUILD → Extraction → Candidates(+Evidence)
④ 검수(승인/수정/반려)      REVIEW → Review → 워크벤치
⑤ 게시(자격 통과분 게시)    PUBLISH → Publish → Published Graph(탐색기/시각화·요약)
⑥ 품질·거버넌스로 개선/통제 REVIEW → Quality / Governance(+영향 시뮬레이션)
⑦ 활용                      ANALYZE → Search / RAG / Copilot / Benchmark …
        ↺ 반복하며 그래프를 키우고 품질을 높임
```

이 가이드는 실제 구동 화면(대시보드·프로젝트 홈·온톨로지·소스·추출·후보·검수·게시 큐·게시 그래프·품질·거버넌스·코파일럿)을 따라 작성되었습니다. 문구는 지속적으로 다듬어지므로 라벨이 조금 달라도 **구축 → 검수 → 게시 → 활용** 흐름은 동일합니다.
